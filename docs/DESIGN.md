# 競艇 AI 自動売買システム 設計書

**作成日**: 2026-04-05  
**ステータス**: 設計フェーズ（未実装）

---

## 1. システム概要

本設計書は、既存の競艇 AI 予測システムを拡張し、  
**レーススケジュール監視 → 予測 → バリューベット判断 → 投票 → 結果集計 → フィードバック**  
を全自動で実行するパイプラインの構成を定義する。

### 1.1 スコープ

| 対象 | 内容 |
|------|------|
| 投票サービス | テレボート（BOAT RACE 公式インターネット投票） |
| 対象券種 | 3連単のみ |
| 対象場 | 全24場（設定で絞り込み可） |
| 1レースあたり予算 | 設定ファイルで定義（デフォルト1,000円） |
| 動作環境 | Windows/Linux 常時稼働マシン または VPS |

## 2. アーキテクチャ全体図

```
┌─────────────────────────────────────────────────────────────┐
│                       Orchestrator                          │
│              (スケジューラ / メインループ)                   │
└──────┬──────────┬──────────┬──────────┬───────────┬────────┘
       │          │          │          │           │
       ▼          ▼          ▼          ▼           ▼
  [Scheduler] [Scraper]  [Predictor] [Bettor]  [Recorder]
  レース日程   出走表・     予測エンジン  テレボート   結果収集・
  監視        オッズ取得   (既存)      投票自動化   損益記録
                                          │
                                          ▼
                                    [Notifier]
                                    LINE / Slack
                                    通知
```

将来的には `Predictor` と `Bettor` の間に **ValueModel** を追加し、
予測確率・オッズ・EV・レース条件から「この買い目の EV は信用できるか」を再評価する。

```
[Predictor] → 120通りの予測確率
      ↓
[ValueModel] → adjusted EV / expected return
      ↓
[RiskManager] → 点数上限・1点上限・日次損失上限
      ↓
[Bettor]
```

## 3. コンポーネント詳細

### 3.1 Orchestrator（オーケストレーター）

**役割**: 全コンポーネントを統括する中央制御。

**処理フロー（1レース分）**:

```
[T-60分] レース存在確認 & 出走表取得
[T-10分] 直前情報 & オッズ取得 → 予測実行 → EV計算
[T- 5分] バリューベット判断 → 投票指示
[T+ 5分] 締切確認（投票完了チェック）
[T+30分] レース結果取得 → 損益計算 → ログ記録 → 通知
[翌日  ] 日次レポート生成
```

**監視対象の決定**:
- 毎朝 8:00 に当日の全場・全レース番号をスクレイピング
- フィルタ条件（設定可）: 場、グレード、開催日次など
- 当日スケジュールをキューに積む

---

### 3.2 Scheduler（スケジューラ）

**役割**: レース発走時刻に合わせて処理をトリガーする。

**実装方式の選択肢**:

| 方式 | メリット | デメリット |
|------|----------|------------|
| `APScheduler`（Python） | 単一プロセスで完結 | 再起動で状態消失 |
| `cron` + スクリプト | OS標準、堅牢 | 柔軟な時刻制御が難 |
| `Celery` + `Redis` | 分散・再試行対応 | 構成が複雑 |

**推奨**: `APScheduler` + SQLite ジョブストア（状態永続化）

**スケジューリングロジック**:
- boatrace.jp の番組表ページ（`https://www.boatrace.jp/owpc/pc/race/index`）を定期ポーリング
- 発走予定時刻の **10分前** に予測・投票ジョブをキューに追加

---

### 3.3 Scraper（スクレイパー）

**役割**: レーススケジュール・出走表・オッズ・結果を取得する。

**既存実装**: `data/scraper.py`（出走表・オッズ）

**追加が必要な関数**:

| 関数名 | 取得先 URL | 用途 |
|--------|-----------|------|
| `scrape_schedule(date)` | `/owpc/pc/race/index?hd=YYYYMMDD` | 当日全レース一覧 |
| `scrape_result(jcd,hd,rno)` | `/owpc/pc/race/raceresult?...` | レース結果・払戻金 |
| `scrape_balance()` | テレボート残高ページ | 現在の口座残高確認 |

---

### 3.4 Predictor（予測エンジン）

**役割**: スクレイピング済みデータから EV 付き投票候補リストを生成。

**既存実装**: `predict.py` / `app.py`の `run_prediction()`

**変更点**:
- Web UI への描画処理を切り離し、**純粋な辞書/データクラスを返す関数**として分離
- キャリブレータのロードをプロセス起動時に1回だけ行うよう最適化

**出力形式（例）**:
```python
{
  "race_id": "04-20260405-08",
  "bets": [
    {"combo": "1-3-5", "amount": 300, "ev": 0.42, "prob": 0.031, "odds": 46.5},
    {"combo": "3-1-5", "amount": 100, "ev": 0.18, "prob": 0.021, "odds": 56.2},
  ],
  "total_bet": 400,
  "bet_mode": "value",
  "confidence": 28.3,
}
```

### 3.4.1 ValueModel（価値判定モデル / 将来拡張）

**役割**: 既存モデルの予測確率とリアルタイムオッズから、買い目ごとの実質的な期待回収額を推定する。

現状の EV は以下で計算している。

```text
raw_ev = model_prob * odds - 1
```

しかし、`model_prob` の誤差や Plackett-Luce の結合確率誤差により、
raw EV がプラスでも実際には負けるケースが多い。  
そのため、買い目単位のメタモデルで raw EV を補正する。

**初期実装候補**:

| 方式 | 用途 | 優先度 |
|------|------|--------|
| LightGBM Regressor | 100円あたり期待回収額を予測 | 高 |
| LightGBM Ranker | レース内の買い目を収益性順に並べる | 中 |
| LightGBM Classifier | レース参加可否、買い目採用可否を分類 | 中 |
| Set Transformer / DeepSets | 6艇集合と120買い目の関係を深層学習 | 低 |

**入力特徴量例**:

- 既存モデルの3連単確率
- オッズ
- raw EV
- 予測確率順位
- 人気順位
- 予測順位と人気順位のズレ
- 1号艇の過剰人気度
- オッズ分布の集中度
- 場、レース番号、天候、風速、波高
- 戦略別の候補数・推定投資額

**目的変数例**:

```text
target_return = 的中した場合は odds、外れた場合は 0
adjusted_ev = predicted_return - 1
```

ValueModel は「買う候補」を決めるために使い、金額配分は RiskManager と capped Kelly / dutching / flat で制御する。
モデルに金額まで直接出させると過学習時の一点集中リスクが高いため、投資上限は必ずルールで縛る。

### 3.5 Bettor（投票自動化）

**役割**: テレボートへログインし、指定された組み合わせ・金額で投票する。

**テレボート接続方式**:

テレボートは公開 API を提供していない。以下の2方式を検討する。

#### 方式A: Playwright / Selenium によるブラウザ自動化（推奨）

```
ログイン → 場/レース選択 → 3連単タブ → 
組み合わせ入力 → 金額入力 → 確認 → 投票実行
```

**メリット**:
- 公式サイトの UI 変更がなければ安定動作
- ヘッドレスモードで無人動作可能

**デメリット**:
- UI 変更時にメンテナンスが必要
- CAPTCHA が導入された場合は対応不可

#### 方式B: HTTP リクエスト直接送信

テレボートのログイン後セッションを取得し、投票フォームのリクエストを直接送信。

**メリット**: 高速・軽量  
**デメリット**: セッション管理が複雑、利用規約の確認が必要

#### セキュリティ要件

- テレボートの ID / パスワードは `.env` ファイルに格納し **git 管理外**
- `TELEBOAT_USER` / `TELEBOAT_PASS` / `TELEBOAT_PIN` 環境変数で管理
- 投票前に残高確認を行い、予算の **2倍以上** の残高がない場合は当日の投票を停止

---

### 3.6 RiskManager（リスク管理）

**役割**: 投票実行前の最終安全チェック。

**チェック項目**:

| チェック | 条件 | アクション |
|----------|------|------------|
| 日次損失上限 | 当日損失 > 設定上限（例: 5,000円） | 当日の投票を全停止 |
| 口座残高 | 残高 < 投票額 × 2 | 投票スキップ |
| 連続見送り | 直近 N レース全見送り | アラート通知 |
| オッズ急変 | 投票直前にオッズを再取得し EV が閾値を下回る | 自動キャンセル |
| 最大賭け金 | 1組み合わせ > 上限（例: 500円） | 上限にクリップ |

**設定ファイル例** (`config.yaml`):
```yaml
risk:
  budget_per_race: 1000       # 円
  max_daily_loss: 5000        # 円
  min_ev_threshold: 0.05      # 5%
  kelly_fraction: 0.25
  min_bet: 100                # 円
  max_bet_per_combo: 500      # 円
  min_balance_ratio: 2.0      # 残高 / 投票額の最低比率

filter:
  venues: []                  # 空 = 全場, 例: ["04", "12"]
  grades: []                  # 空 = 全グレード
  max_races_per_day: 20
```

### 3.7 Recorder（結果記録）

**役割**: 投票履歴・レース結果・損益を SQLite に記録する。

**DB スキーマ（概略）**:

```sql
-- レース単位の予測・投票記録
CREATE TABLE bets (
    id          INTEGER PRIMARY KEY,
    race_id     TEXT NOT NULL,       -- "04-20260405-08"
    combo       TEXT NOT NULL,       -- "1-3-5"
    amount      INTEGER NOT NULL,    -- 円
    odds_at_bet REAL,                -- 投票時オッズ
    ev_at_bet   REAL,                -- 投票時EV
    prob        REAL,                -- モデル予測確率
    status      TEXT DEFAULT 'pending',  -- pending/win/lose
    payout      INTEGER DEFAULT 0,   -- 払戻金（円）
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 日次サマリー
CREATE TABLE daily_summary (
    date        TEXT PRIMARY KEY,
    races_bet   INTEGER,
    races_skip  INTEGER,
    total_bet   INTEGER,
    total_payout INTEGER,
    profit      INTEGER,
    roi_pct     REAL
);
```

### 3.8 Notifier（通知）

**役割**: 投票実行・結果・異常をリアルタイム通知する。

**通知チャネルの選択肢**:

| チャネル | 実装方法 | 用途 |
|----------|----------|------|
| LINE Notify | `requests.post` | モバイル通知（推奨） |
| Slack | Incoming Webhook | PC 通知 |
| メール | `smtplib` | 日次レポート |

**通知タイミング**:
- 投票実行時: 「レース名 / 投票組み合わせ / 金額 / EV」
- レース結果: 「的中/外れ / 払戻金 / 損益」
- 日次サマリー: 「賭けたレース数 / 総損益 / 回収率」
- 異常発生時: 「ログイン失敗 / 残高不足 / 投票エラー」

## 5. 実装優先度とフェーズ

### Phase 1: 監視・通知のみ（投票なし）

投票はせず、「このレースで買うべきだった」通知を出すだけ。  
→ システムの精度・タイミングを人間が確認できる。

| タスク | 難易度 |
|--------|--------|
| `scrape_schedule()` 実装 | 低 |
| Orchestrator（通知のみ） | 低 |
| LINE Notify 通知 | 低 |
| 結果スクレイピング | 中 |
| SQLite 損益記録 | 低 |

### Phase 2: 手動確認付き半自動投票

通知を受けてユーザーが承認した場合のみ投票を実行。

| タスク | 難易度 |
|--------|--------|
| Playwright によるテレボート自動化 | 高 |
| RiskManager 実装 | 中 |
| 承認ボタン付き LINE 返信 | 中 |

### Phase 3: 全自動投票

Phase 2 が安定稼働したら承認ステップを削除して全自動化。

| タスク | 難易度 |
|--------|--------|
| 設定パラメータの最適化 | 中 |
| 異常系ハンドリング強化 | 中 |
| 日次レポート自動生成 | 低 |

## 7. リスク・注意事項

### 技術リスク

| リスク | 対策 |
|--------|------|
| テレボートの UI 変更 | Playwright セレクタをデータ属性ベースにし壊れにくくする |
| スクレイピング失敗 | リトライ機構（最大3回、指数バックオフ）|
| プロセスクラッシュ | systemd / PM2 による自動再起動 |
| 時計ずれ | NTP 同期を前提、発走時刻の 30秒前までに投票完了 |

### 運用リスク

| リスク | 対策 |
|--------|------|
| 回収率が期待を下回る | Phase 1 で最低1ヶ月検証してから Phase 2 へ |
| 残高枯渇 | 日次損失上限 + 残高比率チェック |
| テレボートの利用規約違反 | 利用規約を確認の上、自己責任で運用 |
| 個人情報漏洩 | `.env` を git 管理外、ログにパスワードを含めない |

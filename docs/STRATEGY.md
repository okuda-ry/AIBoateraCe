## 2. なぜ勝てないのか — 構造的問題の分析

### 問題A: モデルが「ランキング精度」を最適化しているが「収益」を最適化していない

LambdaRank は「1着艇を正しく予測する順序」を学習する。  
しかし競艇で利益を出すには **「市場（オッズ）が過小評価している組み合わせ」** を見つける必要がある。

```
例:
  モデル: 1号艇 1着確率 60%（正しい）
  オッズ: 1-2-3 = 4.0倍（= 市場の暗示確率 25%）
  
  EV = 0.6 × 4.0 - 1 = +1.4（+140%）  ← これが理想
  
  実際: 人気レースでは1-2-3 = 1.5倍が多く
  EV = 0.6 × 1.5 - 1 = -0.1（-10%）  ← 負け
```

**問題の本質: 強い艇を当てることと、オッズに対して優位性を持つことは別の話。**

### 問題B: EV計算の精度が低い

EV = 予測確率 × オッズ − 1 の「予測確率」精度が不十分。

- Calibration で Brier スコアは改善したが、**3連単の結合確率**（Plackett-Luce）の精度は未検証
- Plackett-Luce モデルは「独立性仮定」を置いているが、実際には **インコース有利・モーター差** などの相関がある

### 問題C: 学習データに「稼げる情報」が欠けている

現在の特徴量はすべて **出走表から取得できる公開情報**。  
市場参加者（他の投票者）も同じ情報を持っており、  
**情報優位性がない → モデルがオッズを上回る確率が低い**。

| 現在の特徴量 | 問題 |
|-------------|------|
| 全国勝率・当地勝率 | 誰でも知っている → オッズに織り込み済み |
| モーター・ボート2連対率 | 同上 |
| 展示タイム | 競艇新聞にも掲載 → 織り込み済み |
| 今節平均ST | 比較的マイナーだが、ST中心レースは専門家も注目 |

### 問題D: サンプル数・テラ銭の壁

3連単のテラ銭は **約25%**。  
EV がプラスになるには、モデルが **市場より25%以上正確** でなければならない。  
これは非常に高いハードルであり、わずかな優位性では吸収されてしまう。

## 3. 勝つための戦略 — 3つのアプローチ

### アプローチ1: 「市場が間違えやすい場面」に絞る（最優先）

全レースに賭けるのではなく、**市場（オッズ）が系統的に間違えやすいパターン**を探す。

**具体的な絞り込み条件の仮説:**

| 条件 | 根拠 |
|------|------|
| 1号艇が人気薄のレース | 競艇ファンは1号艇を過大評価する傾向 → 2号艇以下が割安になりやすい |
| 展示タイムで逆転がある | 番組表では弱く見えるが展示で強い艇 → 市場への反映が遅れる |
| 荒れやすい気象条件（波高高・強風） | 実力差が縮まる → 人気薄に価値が生まれる |
| 後半レース（9R以降）| 前半との比較でオッズ形成が甘くなることがある |
| 地方開催・マイナー場 | 投票額が少なく市場効率が低い可能性 |

**実装方法: レース選択フィルタ**

全レースに賭けず、スコアリングモデルでレースを選別する。

```python
def should_bet_race(race_features, probs_120, odds_dict) -> bool:
    """このレースに賭けるべきか判定する"""
    # 1. オッズの偏り（ジニ係数）が大きすぎるレースは除外
    # 2. 展示タイム逆転が発生しているレースを優先
    # 3. 気象条件フィルタ
    # 4. モデルの信頼度が一定以上
    pass
```

### アプローチ2: モデルに「市場情報」を加える

**オッズ自体を特徴量として学習させる。**

```
現在: 出走表特徴量 → スコア → EV計算
改善: 出走表特徴量 + オッズ暗示確率 → スコア → EV計算
```

オッズの暗示確率 = `1 / (odds × 1.25)`（テラ銭補正）

これを艇ごとの特徴量として加えることで、  
**「市場が過小評価している艇」を直接学習できる。**

ただし注意点:
- 学習時もオッズが必要になる（締切前オッズの過去データが必要）
- スクレイパーで過去オッズを遡って取得するのは困難

**現実的な代替案: 暗示確率との乖離を特徴量にする**

```python
# 暗示1着確率（マーケットの予想）
implied_1st_prob = 1 / (odds_1st * 1.25)

# モデル予測との乖離
model_vs_market = model_prob_1st - implied_1st_prob
# これが大きい艇 = モデルが市場より高く評価している艇
```

### アプローチ3: 損失関数を「収益」に直結させる

現在の LambdaRank は NDCG（ランキング精度）を最適化する。  
**収益率を直接最適化する損失関数に変更する。**

```
現在: minimize( -NDCG )
改善: minimize( -回収率 ) または minimize( -EV )
```

**具体的な方法:**

#### 方法A: カスタム損失関数（LightGBM カスタム objective）

```python
def profit_objective(y_true, y_pred):
    """
    払戻重みを確率の二乗誤差に乗じた損失。
    高配当（= 市場が見逃した）を正確に予測することを重視。
    """
    # payout_weight: 払戻金が大きいレースほど重み大
    loss = payout_weight * (y_true - softmax(y_pred)) ** 2
    return grad, hess
```

#### 方法B: 2段階学習

1. Stage1: LambdaRank でランキングを学習（現状維持）
2. Stage2: Stage1 の予測確率を入力として、**「オッズに対して値がいいか」を2値分類**で学習

```
入力: [model_prob, odds, model_prob × odds - 1 (EV), ...]
出力: この組み合わせが「実際に的中するかどうか」
```

## 4. 即効性のある改善施策（短期）

実装コストが低く効果が見込める改善から着手する。

### 施策1: レース選択フィルタの実装

```python
# 現状: 全レースに予測・ベット判断
# 改善: 「荒れそうなレース」「モデル信頼度が高いレース」のみ対象

filters = {
    "min_confidence": 0.30,     # 1着予測艇の確率が30%以上
    "min_wave": 5,              # 波高5cm以上（荒れる条件）
    "exclude_1st_odds_under": 2.0,  # 1号艇が2倍未満の鉄板レースは除外
}
```

### 施策2: EV計算の精度向上

現在の Plackett-Luce は独立性を仮定しているが、  
**条件付き確率をより正確に推定する。**

```
1着確率: p1, p2, ..., p6
現在: P(1-2-3) = p1 × p2/(1-p1) × p3/(1-p1-p2)   ← 独立性仮定
改善: P(2着|1着=1) を直接学習する条件付きモデルを追加
```

### 施策3: バックテストによる最適パラメータ探索

現在の `min_edge=0.05` `kelly_frac=0.25` がベストか検証されていない。

```python
# バックテストで最適な閾値を探索
for min_edge in [0.0, 0.05, 0.10, 0.15, 0.20]:
    for kelly_frac in [0.10, 0.25, 0.50]:
        roi = backtest(min_edge=min_edge, kelly_frac=kelly_frac)
        print(f"min_edge={min_edge}  kelly={kelly_frac}  ROI={roi:.1f}%")
```

### 施策4: アンサンブル（複数モデルの多数決）

LightGBM 単体ではなく、複数のモデルの予測を平均する。

```
モデルA: 全期間データで学習（汎化性能重視）
モデルB: 直近2年データで学習（最新傾向を重視）
モデルC: 場ごとに特化したモデル（当地特性を学習）
アンサンブル確率 = (A + B + C) / 3
```

## 5. 購入判断を機械学習化する設計

現状の購入判断は `EV = 予測確率 × オッズ - 1` をそのまま使っている。  
しかし、実際のドライランでは大きく負ける日が多く、問題は「着順予測」よりも
**EV が本当に信用できるかを見極める部分** にある。

そのため、いきなり深層強化学習で買い目と金額を直接出すのではなく、
まずは **EV 補正モデル（Bet Value Model）** を導入する。

### 5.1 推奨アプローチ: EV 補正モデル

既存の LightGBM 予測を Stage 1 とし、買い目単位の Stage 2 モデルを作る。

```
Stage 1:
  出走表・直前情報
    → 6艇スコア
    → Plackett-Luce
    → 120通りの3連単予測確率

Stage 2:
  予測確率 + オッズ + raw EV + レース特徴量
    → 100円あたりの期待回収額
    → adjusted EV

Bet sizing:
  adjusted EV > 閾値 の買い目だけを候補化
  capped Kelly / dutching / flat で資金配分
```

Stage 2 はまず LightGBM Regressor / Ranker が第一候補。  
データ量が十分に増えるまでは、深層学習より GBDT 系の方が扱いやすく、過学習も制御しやすい。

### 5.2 学習データの単位

1レースを120行に展開し、1行を1つの3連単買い目として扱う。

| カラム | 説明 |
|--------|------|
| `race_id` | レースID |
| `combo` | 3連単組み合わせ |
| `model_prob` | 既存モデルの予測確率 |
| `odds` | 投票判断時点のオッズ |
| `raw_ev` | `model_prob * odds - 1` |
| `prob_rank` | 予測確率順位 |
| `odds_rank` | 人気順位 |
| `rank_gap` | 予測順位と人気順位のズレ |
| `top1_prob` | 1着最有力艇の確率 |
| `odds_concentration` | オッズ分布の集中度 |
| `venue` / `rno` / `weather` / `wind` / `wave` | レース条件 |
| `target_return` | 的中ならオッズ、外れなら0 |

目的変数は、まず以下を推奨する。

```text
target_return = 的中した場合は odds、外れた場合は 0
```

モデル出力を `predicted_return` として扱い、

```text
adjusted_ev = predicted_return - 1
```

で購入判断に使う。  
単純な的中/不的中分類は外れが多すぎてクラス不均衡が強いため、最初の目的関数としては回帰の方が自然。

### 5.3 買い目ランキングモデル

次の段階では、120通りをレース内グループとして LightGBM Ranker に渡し、
「このレースの中で期待回収が高い買い目」を並べ替える。

```text
group = race_id
label = target_return または profit_per_100yen
features = model_prob, odds, raw_ev, rank_gap, race_features...
```

これは現在の LambdaRank を「艇の着順」ではなく「買い目の収益性」に適用する設計であり、
既存システムとの相性が良い。

### 5.4 レース選択モデル

買い目単位の価値判定とは別に、レース単位で「このレースは参加すべきか」を判定するモデルも有効。

入力例:

- 最有力艇の確率
- 予測上位と人気上位のズレ
- オッズ分布の偏り
- 1号艇の過剰人気度
- 風速・波高・天候
- 場・レース番号
- EV+ 候補数
- 戦略別の推定投資額

出力:

```text
bettable_race_score = このレースに参加すべき度合い
```

最初はルールベースのフィルタで始め、ドライランが蓄積したら LightGBM Classifier / Regressor に置き換える。

### 5.5 深層学習を使う場合

深層学習を使うなら、いきなり強化学習で金額を出すより、以下の構造が現実的。

```text
6艇特徴量
  → DeepSets / Set Transformer
  → レース表現

120買い目特徴量
  → combo embedding
  → オッズ・EV特徴量と結合
  → 各買い目の期待回収額を出力
```

競艇は1レース6艇の集合問題なので、艇の並びに強く依存しすぎない Set 系アーキテクチャが合う。
ただし実装・検証コストが高いため、最初の実装候補ではない。

### 5.6 金額配分は完全には学習させない

金額までモデルに直接出させると、過去データへの過剰適合で一点集中しやすい。  
そのため、ML の責務とルールの責務を分ける。

| 領域 | 担当 |
|------|------|
| 買い目候補の価値判定 | ML / LightGBM |
| 候補の順位付け | ML / Ranker |
| 1レース上限 | ルール |
| 1点上限 | ルール |
| 日次損失上限 | ルール |
| 最低購入単位 | ルール |
| 配分比率 | capped Kelly / dutching / flat |

この分離により、モデルが外した日に破滅的な張り方をするリスクを抑える。

## 6. システム評価メモ

現時点の総合評価は **72 / 100 点** 程度。

| 観点 | 評価 | コメント |
|------|------|----------|
| システム設計・実装力 | 80〜85点 | データ取得、予測、オッズ、戦略比較、モニタリングまで閉じたループがある |
| Webアプリとしての完成度 | 75点前後 | 主要導線は動くが、戦略比較や履歴UIは継続改善余地あり |
| 予測研究基盤としての価値 | 80点前後 | ドライランとDB記録により改善サイクルを回せる |
| 収益システムとしての完成度 | 45〜55点 | 実損益はまだ不安定で、レース選択とEV補正が未成熟 |

評価の要点:

- 着順予測・オッズ連携・モニタリング基盤はかなり良い。
- 一方で、収益性は「当てる力」ではなく「市場より割安な買い目を見つける力」に依存する。
- 現状はランキング精度に寄っており、EV の信頼性検証と買い方の学習化が次の主戦場。
- 勝てるシステムとして判断するには、最低でも月単位のドライランと条件別分析が必要。
